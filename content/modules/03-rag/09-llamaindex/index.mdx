You have now hand-rolled every primitive: chunking, embedding, BM25, RRF, rerank, query transforms, Ragas. Time to rebuild 3.3's 50-line naive RAG in 15 lines of LlamaIndex — and then watch three defaults bite you.

## 50 lines → 15

```python file=naive_3_3.py  // 3.3 — for reference
# load arxiv/*.txt
texts, arxiv_ids = [], []
for p in sorted(ARXIV_DIR.glob("*.txt")):
    t = p.read_text()
    for i in range(0, len(t), 500):
        texts.append(t[i:i+500]); arxiv_ids.append(p.stem)

# embed + normalize
E = np.array(vo.embed(texts, model="voyage-3", input_type="document").embeddings)
E /= np.linalg.norm(E, axis=1, keepdims=True)

# retrieve
def retrieve(q, k):
    qe = np.array(vo.embed([q], model="voyage-3", input_type="query").embeddings[0])
    qe /= np.linalg.norm(qe)
    return [arxiv_ids[i] for i in np.argsort(-(E @ qe))[:k]]

# generate
r = client.messages.create(model="claude-sonnet-4-6", ..., messages=[...])
```

```python file=llamaindex_3_9.py  // 3.9 — same behaviour, fewer lines
from llama_index.core import Settings, SimpleDirectoryReader, VectorStoreIndex
from llama_index.llms.anthropic import Anthropic
from llama_index.embeddings.voyageai import VoyageEmbedding

Settings.llm = Anthropic(model="claude-sonnet-4-6")
Settings.embed_model = VoyageEmbedding(model_name="voyage-3")

docs = SimpleDirectoryReader(str(ARXIV_DIR)).load_data()
index = VectorStoreIndex.from_documents(docs)
engine = index.as_query_engine(similarity_top_k=5)

print(engine.query("What is chain-of-thought prompting?"))
```

Same corpus, same model, same embedding. Fewer characters, more magic. Which is why we spend the rest of the lesson on the magic.

## Three defaults that will bite you

### 1. `SimpleDirectoryReader` → `SentenceSplitter` hides your chunker

`VectorStoreIndex.from_documents` chunks internally with `SentenceSplitter(chunk_size=1024, chunk_overlap=20)`. On arXiv abstracts this gives ~1 chunk per paper — benign. On a 10-K filing, the sentence splitter crosses `ITEM 1A. RISK FACTORS` headings and produces chunks that mix sections. Your 3.5 structural chunker never does this. To match 3.5 behavior you write a custom `NodeParser` that respects headings — which is the code you were hoping to skip.

```python file=override_parser.py
from llama_index.core.node_parser import MarkdownNodeParser
index = VectorStoreIndex.from_documents(docs, transformations=[MarkdownNodeParser()])
```

### 2. `QueryFusionRetriever(mode="simple")` averages raw scores

```python file=fusion_bug.py
from llama_index.retrievers.bm25 import BM25Retriever
from llama_index.core.retrievers import QueryFusionRetriever

bad  = QueryFusionRetriever([dense, bm25], mode="simple",             similarity_top_k=10)
good = QueryFusionRetriever([dense, bm25], mode="reciprocal_rerank",  similarity_top_k=10)
```

`mode="simple"` averages raw scores. BM25 returns unbounded tf-idf magnitudes, cosine returns [0, 1]. The exact scale bug 3.6's RRF section warned against. `mode="reciprocal_rerank"` is RRF — what you actually want. The default is the wrong one.

### 3. `response_mode="compact"` silently truncates the context window

This is the nastiest. The default query engine concatenates all retrieved nodes into a single prompt. When the concatenated context exceeds the model's input limit, LlamaIndex silently truncates the middle of the buffer. No error, no warning. Faithfulness collapses; generation is fluent; retrieval looks fine. You spend hours debugging the generator while the bug is in the synthesizer.

```python file=response_mode_fix.py
engine = index.as_query_engine(
    similarity_top_k=5,
    response_mode="tree_summarize",   # recursive — no silent truncation
)
```

`tree_summarize` synthesizes pairwise up a tree; `refine` synthesizes sequentially. Both scale across many nodes without dropping any. Always instrument the actual post-synthesis prompt length on a few eval examples before trusting the default.

<Callout kind="break" title="Watch it break">
Run `engine.query(very_long_query)` on a corpus where the top-5 nodes sum to >30k tokens. Print `engine.last_response.source_nodes` and compare to what actually made it into the LLM call via a Langfuse trace. With `response_mode="compact"` you will see nodes 2-4 truncated in the middle; with `tree_summarize` all five are summarized and folded. This is why observability (lesson 5.5) is not optional — frameworks lie by omission.
</Callout>
