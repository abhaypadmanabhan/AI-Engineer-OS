Dense retrieval is fuzzy. BM25 is literal. Combine both, then rerank the top 50. Measure the lift.

Cohere is now your default *embedder* (lesson 3.1) AND your paid *reranker* — same SDK, same key, same rate-limit budget. The free fallback for rerank stays `bge-reranker-v2-m3` (local, CPU, no key); using one vendor for both is fine and what most production stacks land on.

<Callout kind="info" title="Trial tier ≠ usable throughput">
Production embedders require a payment method to unlock usable throughput. Trial tiers (Cohere's 100K TPM, Voyage's 10K TPM) exist for API surface testing, not for indexing real corpora — embedding even 100 SEC 10-Ks at trial caps takes 3+ hours and burns the monthly call quota. Plan for paid-tier signup when scoping infrastructure, even if billed usage stays inside the free credit allotment; the rate limits change, not the pricing. This is one of those things tutorials skip.
</Callout>

```python file=bm25_dense.py
import os
import numpy as np, cohere
from rank_bm25 import BM25Okapi
co = cohere.Client(api_key=os.environ["COHERE_API_KEY"])

corpus = [(c, aid) for c, aid in load_chunks()]   # from lesson 3.5's best chunker
texts = [c for c, _ in corpus]

# Dense — cosine on embed-english-v3.0
E = np.array(co.embed(texts=texts, model="embed-english-v3.0", input_type="search_document").embeddings)
E /= np.linalg.norm(E, axis=1, keepdims=True)

# Sparse — BM25 on whitespace-tokenized lowercase
bm25 = BM25Okapi([t.lower().split() for t in texts])

def retrieve(query: str, k: int = 10) -> list[int]:
    qe = np.array(co.embed(texts=[query], model="embed-english-v3.0", input_type="search_query").embeddings[0])
    qe /= np.linalg.norm(qe)
    dense_ranks = np.argsort(-(E @ qe))
    sparse_ranks = np.argsort(-np.array(bm25.get_scores(query.lower().split())))
    return rrf([dense_ranks[:50], sparse_ranks[:50]], k)
```

## Reciprocal rank fusion — 6 lines

```python file=rrf.py
def rrf(rankings: list[np.ndarray], k: int, k_rrf: int = 60) -> list[int]:
    scores: dict[int, float] = {}
    for r in rankings:
        for rank, doc_id in enumerate(r):
            scores[int(doc_id)] = scores.get(int(doc_id), 0) + 1 / (k_rrf + rank)
    return sorted(scores, key=scores.get, reverse=True)[:k]
```

RRF has one knob (`k_rrf`, default 60 from the original paper). It does not care about score scales. It is scale-free, language-independent, and the production default you should reach for when someone says "hybrid retrieval".

## Rerank — the second pass (pick one)

### Production (Cohere, paid — $1/1k searches)

```python file=rerank_cohere.py
import cohere
co = cohere.Client()   # COHERE_API_KEY

def rerank_cohere(query: str, candidate_ids: list[int], final_k: int = 10) -> list[int]:
    docs = [texts[i] for i in candidate_ids]
    r = co.rerank(query=query, documents=docs, model="rerank-v3.5", top_n=final_k)
    return [candidate_ids[x.index] for x in r.results]
```

### Local (BAAI/bge-reranker-v2-m3, free, CPU-ok)

```python file=rerank_bge.py
from sentence_transformers import CrossEncoder
bge = CrossEncoder("BAAI/bge-reranker-v2-m3", max_length=512)

def rerank_bge(query: str, candidate_ids: list[int], final_k: int = 10) -> list[int]:
    pairs = [(query, texts[i]) for i in candidate_ids]
    scores = bge.predict(pairs)    # CPU: ~30ms per pair on a modern laptop
    order = sorted(range(len(candidate_ids)), key=lambda i: -scores[i])[:final_k]
    return [candidate_ids[i] for i in order]

# Full pipeline — identical shape, swap one line
cands = retrieve(query, k=50)        # RRF fuse to 50
final = rerank_bge(query, cands, 10) # local reranker
```

Both reranker types run a cross-encoder over (query, doc) pairs — slower but far more accurate than bi-encoder dense retrieval. Only feasible on a shortlist, which is why retrieval → rerank is a two-stage funnel in every production system.

**Quality delta (measured on the shipped 20-Q arXiv eval):** Cohere rerank-v3.5 lands nDCG@10 ≈ 0.86; bge-reranker-v2-m3 ≈ 0.83. ~3 points of nDCG is the premium you pay for the hosted endpoint — meaningful at legal/medical-grade faithfulness, rounding error for prototypes. Free tier is never the *better* option; it is the *cheaper* option that is good enough.

## Metadata pre-filter — the single most common production fix

Hybrid retrieval is not enough when your corpus contains multiple entities. The query "Apple FY2023 R&D" matches "research and development" tokens across every tech filing in your index — half the top-k will come from the wrong company. BM25 and dense both score on text similarity; neither knows about *whose* document a chunk belongs to. The fix is a metadata pre-filter:

```python file=prefilter.py
def hybrid_topk(idx, q, k, allowed_tickers=None):
    mask = None
    if allowed_tickers:
        mask = np.array([c["ticker"] in allowed_tickers for c in idx["chunks"]])
    dense_scores = idx["E"] @ qe
    bm25_scores  = np.array(idx["bm25"].get_scores(q.lower().split()))
    if mask is not None:
        dense_scores = np.where(mask, dense_scores, -np.inf)   # not eligible for top-k
        bm25_scores  = np.where(mask, bm25_scores,  -np.inf)
    # …RRF as before
```

Critical: this is a *pre-filter*, not a post-filter. Non-matching chunks must never compete for top-k — masking to `-inf` before `argsort` is the cheap correct way. Post-filtering (retrieve top-50 then drop wrong-ticker rows) loses recall: when many top-50 are wrong-ticker, you end up with 5 right-ticker chunks instead of the 5 *best* right-ticker chunks.

Where the metadata comes from: routing tier (LLM classifier maps the query to an entity), or — better — a structured query API (the user clicked "AAPL" before typing). Lesson 3.10's mastery gate uses ticker pre-filtering on SEC 10-Ks; we discovered the need empirically when context_precision tanked at 0.44 without it. The most common production RAG fix is not a smarter retriever — it is restricting *what the retriever is allowed to see*.
