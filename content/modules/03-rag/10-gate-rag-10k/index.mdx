## Structural chunking on `## ITEM N.` markers

10-Ks have explicit section anchors (`ITEM 1A. RISK FACTORS`, `ITEM 7. MANAGEMENT'S DISCUSSION AND ANALYSIS`, etc.). The fetcher inserts `## ITEM N. TITLE` markers; the chunker breaks on them. Recursive splitters cross these headers and mix risk disclosures into business overview chunks — structural chunking eliminates that failure class.

## Debugging diary — what Ragas surfaced

First proxy run:

```text
faithfulness     : 0.970   ← pipeline is grounded
context_precision: 0.439   ← retrieval is wrong
answer_relevance : 0.159
context_recall   : 0.369
```

High faithfulness + collapsing ctx_precision is a specific failure shape: the generator is being honest about a lousy context. Faithfulness asks whether the answer is grounded in retrieved chunks; it does not ask whether those chunks were the right ones to retrieve.

Diagnosis: the `expected_tickers` field on each eval question was never reaching the retriever. "Apple R&D" was matching `research and development` tokens across 99 filings; ~half the top-5 came from wrong companies. Fix is a metadata pre-filter — the most common production RAG fix:

```python file=prefilter.py
def hybrid_topk(idx, q, k, allowed_tickers=None):
    mask = (np.array([c["ticker"] in allowed_tickers for c in idx["chunks"]])
            if allowed_tickers else None)
    dense = idx["E"] @ qe
    bm25  = np.array(idx["bm25"].get_scores(q.lower().split()))
    if mask is not None:
        dense = np.where(mask, dense, -np.inf)
        bm25  = np.where(mask, bm25,  -np.inf)
    return rrf([np.argsort(-dense)[:50], np.argsort(-bm25)[:50]], k)
```

Pre-filter, not post-filter. Wrong-ticker chunks must never compete for top-k.

## When a correct fix makes the score drop

Three iterations of the proxy run:

```text
                  run 1: build   run 2: + ticker filter   run 3: + chunk regex fix
faithfulness     :    0.970          0.961                    0.962
context_precision:    0.439          0.346                    0.295
context_recall   :    0.369          0.286                    0.313
answer_relevance :    0.159          0.445                    0.587
```

Each fix did what it was designed to do — pre-filter eliminated 0/50 wrong-ticker chunks in top-3; regex fix turned 6 garbage labels per filing into 16 real ones. Yet ctx_precision dropped each iteration. answer_relevance climbed +0.42 cumulative; faithfulness held at 0.96.

The drop wasn't regression — it was the metric losing tolerance for noise it had silently accepted. Run 1's 0.44 was padded by lucky cross-corpus matches; run 2's 0.35 by lucky alignments inside noisy chunk boundaries; run 3's 0.30 is the true precision of top-5 from a tight pool of ~430 right-ticker chunks where the answer lives in 3-5 of them. A 1-in-86 base rate that semantic + BM25 + RRF lift to ~30%. Past it requires architectural change, not tuning.

## Where we landed

```text
gate     : faithfulness ≥ 0.85 AND ctx_precision ≥ 0.75
proxy    : faithfulness   0.962  ✓    ctx_precision   0.295  ✗
```

Faithfulness met by margin. ctx_precision short by ~0.45 — a structural ceiling, not a bug. Path to closing it (architectural, previews Capstone M2; not stacked in this lesson):

- **Question-aware section routing**: classify question → `Risk Factors | MD&A | Financials`, mask retrieval to matching section + ticker. Two-tier metadata pre-filter, same pattern applied twice. Lift estimate 0.55–0.70.
- **Top_k = 10 + Cohere rerank-v3.5** (3.6 pattern): rerank typically adds +0.05–0.15 on rank-weighted metrics. Combined lift estimate 0.45–0.55.
- **Semantic chunking** (3.5) inside financial sections, or one chunk per line item.

The lesson: **Ragas's four metrics together are a diagnostic, not just a gate**. High faithfulness + low ctx_precision means retriever or chunking, not generator. Correct fix to retrieval can reveal chunking was the bigger bug; correct fix to chunking can reveal a structural ceiling. Expect the second bug; know when the next bug is architectural. The pipeline is complete, runnable, debuggable — gate it does not pass is a design constraint published transparently.

## Routing — three transforms, one pipeline

```python file=route.py
def route(q, qtype):
    if qtype == "comparison":  return _step_back(q)   # 3.7 — abstraction surfaces both filings
    if qtype == "factual":     return [q]             # baseline — literal tokens win
    return _hyde(q)                                   # multi_hop — answer-shaped retrieval
```

Every transform from 3.7 has a place; none is universal. The router is one Sonnet call cheaper than running all transforms in parallel and ~95% as accurate on the labeled dataset.

## Faithfulness ≥ 0.85 AND context_precision ≥ 0.75 — what the gate asks

Both must pass for production. Faithfulness alone is gameable (the generator picks one true sentence and stops); context_precision alone does not enforce grounded generation. The conjunction is the contract. Per SPEC.md L3 gate.

```python file=gate.py
passed = (
    summary.faithfulness_mean      >= 0.85 and   # we hit 0.962 ✓
    summary.context_precision_mean >= 0.75       # we hit 0.295 ✗
)
```
