## Qdrant — the default

```python file=qdrant_demo.py
import numpy as np
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct

client = QdrantClient(url="http://localhost:6333")
client.recreate_collection(
    collection_name="arxiv",
    vectors_config=VectorParams(size=1024, distance=Distance.COSINE),
)

# Insert 1,000 random vectors with a payload filter we'll use later
rng = np.random.default_rng(0)
vecs = rng.standard_normal((1000, 1024)).astype("float32")
vecs /= np.linalg.norm(vecs, axis=1, keepdims=True)
client.upsert("arxiv", points=[
    PointStruct(id=i, vector=vecs[i].tolist(), payload={"category": ["cs.CL","cs.CV","cs.LG"][i % 3]})
    for i in range(1000)
])

# Query with a payload filter — this is where Qdrant shines over pgvector for complex filters
from qdrant_client.models import Filter, FieldCondition, MatchValue
hits = client.search(
    collection_name="arxiv",
    query_vector=vecs[0].tolist(),
    query_filter=Filter(must=[FieldCondition(key="category", match=MatchValue(value="cs.CL"))]),
    limit=5,
)
print([(h.id, round(h.score, 3)) for h in hits])
```

<CostMeter cost="$0.00" latency="3ms" tokens="0" />

Search p95 for 1k vectors at `ef=64`: ~3ms on a laptop. HNSW default. If you flip `distance` to `DOT` and pre-normalize (which we do), search gets ~30% faster and results are identical.

## pgvector — Postgres you already run

```python file=pgvector_demo.py
import numpy as np, psycopg
from pgvector.psycopg import register_vector

conn = psycopg.connect("postgresql://postgres:dev@localhost:5432/postgres", autocommit=True)
conn.execute("CREATE EXTENSION IF NOT EXISTS vector")
register_vector(conn)
conn.execute("DROP TABLE IF EXISTS arxiv")
conn.execute("CREATE TABLE arxiv (id int PRIMARY KEY, category text, emb vector(1024))")
conn.execute("CREATE INDEX ON arxiv USING hnsw (emb vector_cosine_ops)")

rng = np.random.default_rng(0)
vecs = rng.standard_normal((1000, 1024)).astype("float32")
vecs /= np.linalg.norm(vecs, axis=1, keepdims=True)
with conn.cursor().copy("COPY arxiv (id, category, emb) FROM STDIN") as cp:
    for i in range(1000):
        cp.write_row((i, ["cs.CL","cs.CV","cs.LG"][i % 3], vecs[i]))

rows = conn.execute(
    "SELECT id, 1 - (emb <=> %s) AS score FROM arxiv WHERE category='cs.CL' ORDER BY emb <=> %s LIMIT 5",
    (vecs[0], vecs[0]),
).fetchall()
print(rows)
```

Same result, fewer ops. If Postgres is already running your app, this is the fastest path to production. Cosine distance operator `<=>` ships with pgvector; the index uses HNSW just like Qdrant.

## Chroma — prototype speed

```python file=chroma_demo.py
import chromadb, numpy as np
client = chromadb.Client()
col = client.create_collection("arxiv", metadata={"hnsw:space": "cosine"})
rng = np.random.default_rng(0); v = rng.standard_normal((1000, 1024)).astype("float32")
v /= np.linalg.norm(v, axis=1, keepdims=True)
col.add(ids=[str(i) for i in range(1000)], embeddings=v.tolist(),
        metadatas=[{"category": ["cs.CL","cs.CV","cs.LG"][i % 3]} for i in range(1000)])
print(col.query(query_embeddings=[v[0].tolist()], n_results=5, where={"category": "cs.CL"}))
```

In-process, zero services. Perfect for the first naive-RAG lesson. Loses at scale — no built-in sharding, slower than Qdrant past ~500k vectors.

<Callout kind="break" title="Watch it break">
Insert 1M vectors into Chroma's default SQLite backend. Watch memory balloon past 8GB and queries crawl to ~200ms. Chroma isn't wrong — it is tuned for the 10k-vector notebook case. At production scale, you graduate to Qdrant or pgvector. Picking the wrong store for the corpus size is the most common vector-infra mistake.
</Callout>

## Decision matrix

| | Qdrant | pgvector | Chroma | Pinecone |
|---|---|---|---|---|
| Ops burden | docker | none if you have PG | none | none (SaaS) |
| Scale ceiling | 100M+ | 10M comfortable | 500k | unlimited |
| Filter expressivity | rich | SQL (best) | basic | ok |
| Hybrid search | native | via extensions | manual | native |
| Cost at 10M vectors | self-host | self-host | — | ~$70/mo |
| Best for | prod RAG | PG-native apps | prototypes | zero-ops teams |

Default for this course: **Qdrant**. The capstone M2 runs on it. pgvector is shown side-by-side so you can port in 10 lines when your team insists on staying in Postgres.
