Retrieval nDCG is not enough. A RAG pipeline can retrieve the right chunks and still answer the wrong question, or answer it from thin air. Ragas is four LLM-judged metrics that catch different failure modes. We use the `ragas` library on the primary path — battle-tested prompts, maintained across metric revisions — and show the math by hand in two teaching cells so the library is not a black box.

## The four metrics

| Metric | Asks | Catches |
|---|---|---|
| Faithfulness | Are answer claims supported by context? | Hallucination |
| Answer relevance | Does the answer address the question? | Off-topic answers |
| Context precision | Are retrieved chunks relevant, in rank order? | Poor retrieval ranking |
| Context recall | Do the chunks contain the ground-truth info? | Missing retrieval |

## Primary path — ragas with an Anthropic judge

```python file=ragas_anthropic.py
from ragas import EvaluationDataset, SingleTurnSample, evaluate as ragas_evaluate
from ragas.llms import LangchainLLMWrapper
from ragas.embeddings import LangchainEmbeddingsWrapper
from ragas.metrics import (
    Faithfulness, ResponseRelevancy,
    LLMContextPrecisionWithReference, LLMContextRecall,
)
from langchain_anthropic import ChatAnthropic
from langchain_voyageai import VoyageAIEmbeddings

llm = LangchainLLMWrapper(ChatAnthropic(model="claude-sonnet-4-6", temperature=0, max_tokens=1024))
emb = LangchainEmbeddingsWrapper(VoyageAIEmbeddings(model="voyage-3"))

samples = [SingleTurnSample(user_input=q, response=a, retrieved_contexts=c, reference=g)
           for q, a, c, g in zip(questions, answers, contexts, ground_truths)]
result = ragas_evaluate(
    dataset=EvaluationDataset(samples=samples),
    metrics=[Faithfulness(), ResponseRelevancy(),
             LLMContextPrecisionWithReference(), LLMContextRecall()],
    llm=llm, embeddings=emb,
)
df = result.to_pandas()     # per-example scores + means
```

Ragas defaults to OpenAI; the three `Langchain*Wrapper` lines route every judge call through Anthropic and every embedding call through Voyage. Swap `claude-sonnet-4-6` for `claude-haiku-4-5` to cut judge cost ~5× when you are iterating.

```python file=api.py
    questions: list[str], answers: list[str],
    contexts: list[list[str]], ground_truths: list[str],

```

3.10 and Capstone M2 import this signature verbatim. `evaluate()` is the stable contract; the library underneath can be swapped (ragas → custom) without touching caller code.

## Under the hood — faithfulness in ~15 lines

```python file=demo_faithfulness.py
def demo_faithfulness(answer, context, judge_model="claude-sonnet-4-6"):
    claims = extract_atomic_claims(answer, judge_model)          # LLM splits answer
    if not claims: return 1.0
    ctx = "\n\n".join(context)
    supported = sum(
        1 for c in claims
        if nli_judge(ctx, c, judge_model).lower().startswith("yes")   # per-claim NLI
    )
    return supported / len(claims)
```

This is the shape of `ragas.Faithfulness` — split into atomic claims, entail each against the joined context, return the supported fraction. The library's prompts are more carefully tuned (it handles "inferred" vs "stated" claims, etc.); the teaching helper matches within ~0.1 on simple cases. Use it to debug: print the per-claim verdicts when ragas flags a low score.

## Under the hood — context precision

```python file=demo_ctx_precision.py
def demo_context_precision(question, context, judge_model="claude-sonnet-4-6"):
    rel = [1 if ctx_relevant(question, c, judge_model) else 0 for c in context]
    total = sum(rel)
    if total == 0: return 0.0
    # weighted precision@k over rank order — same shape as ragas
    return sum(sum(rel[:k+1]) / (k+1) for k in range(len(rel)) if rel[k]) / total
```

Rewards rank quality: a relevant chunk at rank 1 scores 1.0; at rank 10, scores 0.1. This is why reranking (lesson 3.6) moves this metric directly — not just recall, but order.

## Why one number is not enough

```python file=gotcha.py
q   = "What scaling law governs compute-optimal LLM training?"
ctx = ["Chinchilla: model size and training tokens should scale equally with compute..."]
a_off = "Chinchilla was trained on MassiveText. It uses rotary positional embeddings."
# Every claim is in the context. Faithfulness = 1.0.
# The answer does not mention scaling laws. Answer relevance ~0.3.
```

Faithfulness = 1.0, answer relevance = 0.3. System grounded and useless. Ship only when both are above threshold.

## Eval cost at scale — sampling hygiene

Running all four metrics with a Sonnet judge costs ~$0.02 per example. At 500 questions this is $10 per run; at 5k, $100. You do not run the full suite on every commit. The production pattern: **sample 50 random questions nightly, full suite weekly**, and any drift on the nightly 50 triggers an on-demand full run. Same cadence that Langfuse, Arize, and internal eval stacks converge on — cost is the constraint, coverage is the target. Pin the random seed so nightly comparisons are apples-to-apples.

<Callout kind="break" title="Watch it break">
Use the same model as generator and judge. Self-preference bias: Sonnet judging Sonnet answers rates them ~0.05 higher than Haiku or GPT-5 judging the same outputs. Safe rule: pick a judge from a different model family, or if you must use the same family, pin a smaller model (Haiku) so the judge cannot share the generator's specific failure modes. Never publish Ragas numbers without naming the judge.
</Callout>
